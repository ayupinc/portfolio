Cisco Telephony

Reconstructing calls, agent calls, sessions and activities from native Cisco event data so Power BI and SSRS could report from a trusted operational layer.

Context

The Cisco telephony work started as a reporting reliability problem. After a switchboard change, the existing third-party call reporting tables no longer provided consistent queue times, ring times or call presence.

The immediate requirement was to restore operational reporting, but the underlying issue was that the available Cisco source data did not describe calls in the way managers understood them.

Data Engineering Process

I worked from the native Cisco tables rather than the broken reporting layer. Route Call Detail held one row per routed call, while Termination Call Detail held multiple rows for the segments of that call.

Agent Event Detail held login, logout and status activity, but events were split across Cisco interval boundaries rather than stored as complete activity periods. The engineering task was therefore to convert low-level event records into analytical entities: calls, agent calls, sessions and activities.

The first stage was interpretation. I analysed combinations of Peripheral Call Type, Call Disposition and Call Disposition Flag to identify which TCD rows represented routing, ringing, answering, talking, transfer, monitor and internal-call behaviour. This allowed call segments to be classified before aggregation.

The next stage was reconstruction. Routed calls were rebuilt from RCD and related TCD rows using RouterCallKeyDay and RouterCallKey, which were common keys in both tables. Dialled calls were taken directly from TCD but required separate attribution rules.

Internal calls needed particular care because the operational owner changed depending on whether the call was answered, transferred or conferenced. I built separate logic for Call Data and Agent Call Data so the same source event could support both call-volume reporting and agent-perspective reporting.

Agent sessions were reconstructed from Agent Event Detail. Login and logout events created the session boundary, while status-code events were collapsed into continuous activity periods. This produced one row per session and one row per meaningful activity rather than fragmented interval records.

Validation

I validated the data using comparisons with interval reporting, comparisons with existing third-party and Cisco reports, and real-time analysis of agent activity.

Output Model

The output was a reusable telephony analytics model:

  • Call Data
  • Agent Call Data
  • Agent Session Data
  • Agent Activity Data

These tables became the shared reporting layer for Power BI and SSRS.

Outcome

The outcome was not only replacement reporting. The model supported abandonment and demand reporting, handling-time analysis, Clinical Navigator telephony metrics, six agent-level SSRS reports and session-level activity reporting.

It also removed reliance on the unreliable third-party reporting tables and gave operational users a traceable route from KPI back to source event behaviour.
