Clinical Queue Activity

Building a validated queue episode model from CAD audit tables so clinical teams could understand queue placements, waiting times and actions taken while cases were waiting.

Context

Operational changes required more emphasis on understanding activity in the clinical queue where cases were being handled live by clinicians or pending call-backs.

An analytical structure was required which could answer the questions service managers were actually asking: which clinical teams handled each case, how long did cases wait for an assessment, and what other actions were taken while waiting.

The CAD patient management audit tables recorded team assignments, assessment activity, clinician actions and outcomes, but not continuous episodes. A single incident could generate dozens of audit records across multiple systems without any persistent identifier representing the placement or actions within a team.

Data Engineering Process

First, I created a stable analytical entity that the source system did not provide. A team placement ID represented a unique placement of an incident into a clinical team, capturing incident, team name, placement sequence, timestamp, case priority and category at the point of placement.

Not all team placements in the audit stream represented genuine activity. System housekeeping processes generated transient placements, duplicate transitions, timeout removals and technical routing artefacts that would inflate durations and corrupt path analysis if included.

I introduced validation logic to classify each placement and retained the inclusion or exclusion rationale within the output table, rather than applying opaque filters inside individual reports.

Once validated placements existed, activity from CAD cases, assessment software, questionnaire activity and staff-generated events could be attributed to the correct placement episode. The attribution logic sequenced all events chronologically and assigned each action to the active validated level at that point in time.

The most complex stage was reconstructing start and end times for each queue level. End times had to be inferred from subsequent placements, removals, outcomes and lookback logic, with explicit handling for cases that left and returned to the queue, multiple assessments on the same incident, and missing assessment or staff actions.

This temporal reconstruction became important for time-to-start-assessment KPIs. I engineered an endpoint hierarchy with operational teams, consisting of staff action, assessment start or recorded outcome, to prevent triage records being lost where markers were missing.

Validation

I validated the data using comparisons against the original audit event table, with inclusion and exclusion rationale providing evidence of accuracy, plus real-time analysis of clinician activity and comparisons against other limited built-in reports.

Output Model

The engineering process produced two analytical tables:

  • An episode-based row per validated queue placement, with placement start and end timestamps, priority at placement, validation status and rationale for inclusion or exclusion
  • A row per operational action occurring within a validated placement, linked back to the relevant placement ID and recording action source, type and timestamp

Downstream Power BI reporting was built against these tables rather than raw audit sources, centralising validation rules, temporal calculations and KPI definitions in the SQL transformation layer.

Outcome

The output was a validated, queryable queue episode model. Time-to-assessment KPIs became calculable for the first time, with a defined endpoint hierarchy to handle incomplete workflow marker usage.

Validation rationale was persisted within the model rather than applied dynamically, improving auditability and reproducibility. The model was applied to several outputs and reports including clinical call flow, clinical queue monitoring and KPI reporting.
