Questionnaire Analytics Engineering

Structuring raw CAD questionnaire responses into episode and response tables that could support clinical quality, outcome and pathway reporting.

Context

The questionnaire system is built into the CAD Patient Management System and is used by clinicians to prompt questions and capture responses during telephone assessment.

Data included question text, responses, outcome codes and timestamps, but it was stored as one row per question response rather than one row per assessment episode. A single questionnaire completion could produce dozens of rows, and multiple questionnaires had different structures and branch pathways.

The data also carried quality problems inherited from its collection context: inconsistent free-text responses, response categories with multiple spellings, and question text that differed across questionnaire versions covering the same clinical concept.

What I Did

I designed and built the full transformation pipeline in T-SQL, from raw questionnaire source data through to two structured analytical tables consumed by Power BI dashboards.

This included authoring the episode view, the response cleaning stored procedure and the classification logic, as well as investigating and resolving source data behaviours that were not documented.

I identified a branch loop-back issue while validating branch attribution, traced it to the way the CAD system timestamped repeated pathway selections, and changed the approach from max-based selection to row-number-based selection.

I also identified and documented re-open behaviour, where CAD copies original response timestamps into a new instance, and built detection logic to flag affected episodes rather than leaving the distortion hidden inside duration calculations.

Data Engineering Process

The first stage was episode identification. The source table held no persistent identifier representing a completed questionnaire attempt, so one had to be constructed from the incident identifier, questionnaire set identifier and run number within the incident.

A separate derived field ranked each episode within the incident using a dense rank over start time, providing an episode-in-call sequence useful for tracking activity across a single call.

Branch identification required particular care for questionnaire sets where clinicians navigate different pathways. The branch name was extracted from the response to the pathway selection question, with row-number logic used to capture the first selected branch rather than a later loop-back value.

Re-open detection identified sessions sharing the same incident, questionnaire set, branch and start time, flagging the earlier instance as a potential donor and the later as a re-opened episode.

Response normalisation and text cleaning assigned consistent question identifiers and clean question text to each response. A cleaning pass standardised free-text responses before classification, and unmapped responses were flagged explicitly so coverage could be monitored.

Output Model

The engineering process produced two analytical tables:

  • Episodes, with one row per completed questionnaire episode, generated episode key, sequence, questionnaire set, branch, start and end timestamps, duration, donor and re-opened flags
  • Responses, with one row per cleaned and classified response linked to the episode key, consistent question identifier, clean question text, clean response text and assigned response category

Downstream Power BI reporting was built against these tables rather than the raw source, centralising cleaning and classification logic in the transformation layer.

Outcome

The transformation layer made structured questionnaire reporting possible for the first time. Clinical teams could analyse completed assessment episodes, track questionnaire activity within a call and aggregate responses to the same clinical question across different questionnaire versions.

The branch correction and re-open detection work uncovered source data behaviours that had not been documented and were actively distorting analysis attempted against the raw tables. By surfacing these as explicit flags, the reporting layer remained auditable.
