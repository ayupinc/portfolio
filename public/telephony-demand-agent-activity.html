Telephony Demand & Agent Activity

← Back to Power BI portfolio

Example data shown for portfolio presentation only; figures do not represent actual values from any organisation.

What I Did

  • Created the reporting tables in the analytics engineering layer (see link)
  • Designed the Power BI semantic model
  • Developed DAX measures for KPI, duration, percentile and filter-context reporting
  • Built 5 Power BI reports with 42 pages of telephony demand and agent activity
  • Designed report layouts and provided user guidance for an operational audience
  • Optimised report performance across several large datasets

Context

A high-volume contact centre handling incoming calls and long duration outbound calls across several teams. Around 2 million calls per year in and 500,000 out. Third-party telephony reporting tables provided per-call reporting but became unreliable after a switchboard upgrade and were not able to be updated. Other interval-based reports lacked the required depth for the required per-call and per-agent reporting.

Technologies used were T-SQL Server for the underlying data layer and native tables with Power BI for the semantic model, DAX measures and report canvas, along with SQL Server Reporting Services for some agent activity presentations. The work required a full reconstruction of call and agent activity from the native Cisco tables which is detailed in a separate page here.

Data Model

Source data came from four reporting tables constructed from the above-mentioned analytics engineering work which contained call data for calls and for agents and session and activity data for agents. These provided one row per call, one row per call per agent, one row per session, and one row per activity period respectively and formed the basis of the star schema for the Power BI model. Additional tables included lookups for various call types and agent details. Relationships were built around the calls, the agents and their sessions.

The model supported both high-level summary reporting and granular drill-through to individual call and agent activity records without requiring separate models for each use case. Calculated columns handled activity classification and session completeness flags.

DAX measures supported operational KPI calculations, duration and average calculations, percentage and percentile reporting, and filter-context management for cross-filtering between call and session views.

Report Design

Three main themes were reported: demand and workforce utilisation, key performance indicators and agent activity, with specific layouts prioritising clarity for an operational audience. Pages included:

  • Demand analysis by incoming queue, team and time period to support day to day operations and for workforce planning
  • Answer and abandonment performance
  • Handling time analysis at call and agent level
  • Agent session activity including busy, idle and away time

Additional pages provided:

  • Drill through from various charts to show individual call and session details, used for individual performance or investigations
  • Help pages which explained all metrics presented, especially within the agent performance reporting for clarity on what is being shown. Navigation help pages also improved usage efficiency.
  • Version control pages supported users with changes made and ensuring the most up to date copy

Outcome and Usage

Demand reports are used daily in all areas of operations, KPI reports are used regularly to complete reports and agent activity information is used in all agents' one-to-one sessions and to provide anonymous usage to the organisation.

The reconstruction of individual agent activity reports removed reliance on externally generated outputs, avoiding approximately £60k in additional third-party reporting expansion costs. The reporting structures were built for reuse, providing a stable foundation for future development without external dependencies.
